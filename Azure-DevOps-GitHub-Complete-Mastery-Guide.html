Welcome to Your DevOps Journey

This comprehensive guide will transform you from someone who's never used version control into a professional DevOps practitioner capable of managing enterprise-scale software delivery pipelines.

What Is DevOps?

DevOps isn't a tool - it's a philosophy. Think of it like running a restaurant:

  • Traditional approach: Chefs make food (developers write code), waiters serve it (operations deploy it). They barely talk. Food gets cold. Customers complain.
  • DevOps approach: Chefs and waiters work as one team. Food goes from kitchen to table hot and fast. Everyone wins.

DevOps combines Development (building software) and Operations (running software) into one smooth workflow.

Chapter 1: The Version Control Universe üåç

What Is Version Control?

Imagine you're writing a novel. Without version control, you'd have:

  • Novel_Draft.docx
  • Novel_Draft_v2.docx
  • Novel_Draft_v2_FINAL.docx
  • Novel_Draft_v2_FINAL_ACTUALLY_FINAL.docx
  • Novel_Draft_v2_FINAL_ACTUALLY_FINAL_THIS_TIME_I_SWEAR.docx

We've all been there! Version control solves this mess by:

  • üìö Tracking every change - who, what, when, why
  • ‚Ü©Ô∏è Rolling back mistakes - undo to any point in history
  • üë• Enabling collaboration - multiple people, same file, no conflicts
  • üîÄ Managing versions - work on features independently
  • üîç Understanding history - why was this changed?

üéØ The Core Problem Version Control Solves

Multiple people need to edit the same files without:

  • Overwriting each other's work
  • Losing previous versions
  • Breaking working code
  • Emailing files back and forth
  • Not knowing who changed what

What Is Git?

Git is a distributed version control system. Let's break that down:

  • Version Control System: Tracks changes to files over time
  • Distributed: Everyone has a full copy of the entire history

Think of Git like a time machine for your code. Every time you "commit" (save a snapshot), Git remembers:

  • What files changed
  • What the changes were (line by line)
  • Who made the changes
  • When they were made
  • Why (if you write a good commit message)

Key Git Concepts

Repository (Repo)

A folder that Git tracks. Contains all your project files plus Git's hidden history database (.git folder).

Commit

A snapshot of your repository at a point in time. Like saving your game progress.

Branch

A parallel version of your code. Like working on a copy while keeping the original safe.

Merge

Combining changes from one branch into another.

Clone

Making a local copy of a remote repository.

Push

Sending your local commits to a remote repository.

Pull

Getting changes from a remote repository to your local machine.

üéì The Restaurant Kitchen Analogy

Think of Git like a restaurant kitchen:

  • Repository: The kitchen itself
  • Commit: Completing a dish and putting it on the pass
  • Branch: Different stations (grill, pastry, sauce) working independently
  • Merge: Plating all components together
  • Main/Master branch: The head chef's approved recipes
  • Pull Request: Junior chef asking head chef to approve their new recipe

Configuring Git

Before using Git, tell it who you are:

Chapter 2: Git Fundamentals - Your First Repository üìÅ

Right, no more theory. Time to actually USE Git. By the end of this chapter, you'll have created your first repository, made commits, and understood the basic workflow.

Creating Your First Repository

There are two ways to start: create new or clone existing. We'll start with creating new.

Method 1: Start from Scratch (Local First)

# Create a new folder mkdir my-first-repo cd my-first-repo # Initialize Git git init # Output: Initialized empty Git repository in /path/to/my-first-repo/.git/

That's it! You've created a Git repository. The git init command creates a hidden .git folder that stores all the version history.

üí° What Just Happened?

The git init command created a .git folder in your directory. This hidden folder is Git's database. Everything Git knows about your project's history lives here.

Never delete the .git folder! That's your entire history.

Making Your First Commit

A commit is a snapshot of your project. Let's create one:

# Create a file echo "# My First Repository" > README.md # Check status (see what's changed) git status # Add file to staging area git add README.md # Commit with a message git commit -m "Initial commit: Add README"

Let's break down what we just did:

  1. Created a file: README.md is a markdown file that describes your project
  2. Checked status: git status shows what's changed
  3. Staged the file: git add tells Git "I want to include this in my next commit"
  4. Committed: git commit creates the snapshot with a message explaining what changed

üéØ The Three States of Git

  1. Working Directory: Your files as they are now (untracked changes)
  2. Staging Area (Index): Files ready to be committed
  3. Repository: Committed snapshots (permanent history)

Think: Working Directory = rough draft, Staging = final draft, Repository = published book

Understanding git status

This is your most-used command. It tells you what's going on:

git status # Output shows: # - Which branch you're on # - Untracked files (new files Git hasn't seen) # - Modified files (files Git tracks that have changed) # - Staged files (files ready to commit)

Run git status constantly. Before and after every operation. It's your safety net.

The git add Command

Adding files to the staging area:

# Add a single file git add filename.txt # Add multiple specific files git add file1.txt file2.txt file3.txt # Add all files in current directory git add . # Add all files in entire repository git add -A # Add all files with specific extension git add *.sql

üö® Common Mistake: git add .

Be careful with git add .

This adds EVERYTHING in your current directory, including:

  • Temporary files you don't want
  • Sensitive files (passwords, keys)
  • Large files that shouldn't be in Git

Better: Add files explicitly, or use git add -p to review each change

Writing Good Commit Messages

Commit messages are documentation. Future you (and your team) will read them. Make them good:

‚úÖ Good Commit Messages

# Good - clear and specific git commit -m "Add customer validation to order form" git commit -m "Fix bug in date calculation for leap years" git commit -m "Update README with installation instructions" git commit -m "Refactor database connection logic"

‚ùå Bad Commit Messages

# Bad - vague and useless git commit -m "fixed stuff" git commit -m "update" git commit -m "asdfasdf" git commit -m "hopefully this works" git commit -m "changes"

üí° Commit Message Template

Format: [Action] [What] [Why if not obvious]

  • "Add feature to export reports as PDF"
  • "Fix null reference error in customer lookup"
  • "Remove deprecated API endpoints"
  • "Update dependencies to fix security vulnerability"

Start with a verb: Add, Fix, Update, Remove, Refactor

Viewing Commit History

# Show commit history git log # Show history in one line per commit git log --oneline # Show history with graph (shows branching) git log --oneline --graph --all # Show last 5 commits git log -5 # Show commits by specific author git log --author="Your Name"

Making Multiple Commits

Let's practice the workflow:

# Create a new file echo "print('Hello World')" > hello.py # Check what changed git status # Stage the file git add hello.py # Commit git commit -m "Add hello world Python script" # Modify the file echo "print('Goodbye!')" >> hello.py # Check status (file is modified) git status # Stage and commit in one line git add hello.py git commit -m "Add goodbye message to hello script" # View your history git log --oneline

Undoing Changes

Made a mistake? Git has your back:

Scenario 1: Undo Changes Before Staging

# You modified a file but haven't staged it # Want to discard changes and go back to last commit git checkout -- filename.txt # Or in Git 2.23+: git restore filename.txt

Scenario 2: Unstage a File

# You ran 'git add' but want to unstage git reset HEAD filename.txt # Or in Git 2.23+: git restore --staged filename.txt

Scenario 3: Amend Last Commit

# Forgot to include a file in your last commit # Or want to change the commit message git add forgotten-file.txt git commit --amend -m "New commit message"

‚ö†Ô∏è Warning: Never Amend Pushed Commits!

Only use --amend on commits that are still local. If you've pushed to a shared repository, amending will cause problems for others!

The .gitignore File

Some files shouldn't be tracked by Git. Tell Git to ignore them:

# Create .gitignore file touch .gitignore # Edit it (or use VS Code) nano .gitignore

Common things to ignore:

# .gitignore file contents # Operating system files .DS_Store Thumbs.db # Editor files .vscode/ .idea/ *.swp # Dependencies node_modules/ venv/ __pycache__/ # Build outputs dist/ build/ *.exe # Sensitive files .env config.ini *.key *password* # Logs *.log logs/ # Databases *.db *.sqlite

üí° gitignore Tips

  • Create .gitignore BEFORE first commit
  • Use gitignore.io to generate templates
  • Commit your .gitignore file
  • If you accidentally committed something, removing it from .gitignore won't untrack it - you need git rm --cached filename
